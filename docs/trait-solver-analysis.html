Rust Trait Solver: Integration Analysis for SAGE

An analysis of rustc_next_trait_solver and rustc_type_ir, evaluating whether to integrate them into SAGE vs. writing a custom solver.

1. Architecture: The Three-Layer Cake

The "next-gen" trait solver is split into three crates with a clear dependency hierarchy:

┌─────────────────────────────────────────────────────────────────┐ │ rustc_trait_selection │ │ Rustc-specific: wires the solver to TyCtxt, error reporting, │ │ obligation fulfillment. NOT reusable outside rustc. │ └─────────────────────────────┬───────────────────────────────────┘ │ depends on ┌─────────────────────────────▼───────────────────────────────────┐ │ rustc_next_trait_solver │ │ The actual solver engine. Generic over Interner. │ │ Performs: goal evaluation, normalization, coherence, │ │ candidate assembly, search graph (cycle detection). │ │ ~15k lines. This is the reusable part. │ └─────────────────────────────┬───────────────────────────────────┘ │ depends on ┌─────────────────────────────▼───────────────────────────────────┐ │ rustc_type_ir │ │ Type IR definitions: TyKind, RegionKind, PredicateKind, │ │ Interner trait, InferCtxtLike, folding, visiting, relating. │ │ ~30k lines. The foundation everything builds on. │ └─────────────────────────────────────────────────────────────────┘

The key insight: rustc_next_trait_solver is parameterized over an Interner trait. In theory, any project can implement Interner and plug in. In practice, the Interner trait surface is enormous.

2. The Interner Trait: What You'd Need to Implement

The Interner trait lives at compiler/rustc_type_ir/src/interner.rs. It is the single most critical interface. Here's its shape:

pub trait Interner:
    Sized + Copy
    + IrPrint<ty::AliasTy<Self>>
    + IrPrint<ty::TraitRef<Self>>
    + IrPrint<ty::FnSig<Self>>
    + ... // ~12 more IrPrint bounds
{
    // === ~25 DefId associated types ===
    type DefId: DefId<Self>;
    type TraitId: SpecificDefId<Self>;
    type ImplId: SpecificDefId<Self>;
    type AdtId: SpecificDefId<Self>;
    type TraitAssocTyId: SpecificDefId<Self>;
    type TraitAssocConstId: SpecificDefId<Self>;
    type OpaqueTyId: SpecificDefId<Self>;
    type FreeTyAliasId: SpecificDefId<Self>;
    type InherentAssocTyId: SpecificDefId<Self>;
    // ... many more

    // === Core IR types ===
    type Ty: Ty<Self>;
    type Region: Region<Self>;
    type Const: Const<Self>;
    type GenericArgs: GenericArgs<Self>;
    type GenericArg: GenericArg<Self>;
    type ParamEnv: ParamEnv<Self>;
    type Predicate: Predicate<Self>;
    type Clause: Clause<Self>;

    // === Query methods (the expensive part) ===
    fn generics_of(self, def_id: Self::DefId) -> Self::GenericsOf;
    fn type_of(self, def_id: Self::DefId) -> ty::EarlyBinder<Self, Self::Ty>;
    fn predicates_of(self, def_id: Self::DefId)
        -> ty::EarlyBinder<Self, impl IntoIterator<Item = Self::Clause>>;
    fn item_bounds(self, def_id: Self::DefId)
        -> ty::EarlyBinder<Self, impl IntoIterator<Item = Self::Clause>>;
    fn explicit_super_predicates_of(self, def_id: Self::TraitId)
        -> ty::EarlyBinder<Self, impl IntoIterator<Item = (Self::Clause, Self::Span)>>;
    fn impl_trait_ref(self, impl_def_id: Self::ImplId)
        -> ty::EarlyBinder<Self, ty::TraitRef<Self>>;

    // === Impl enumeration (critical for solving) ===
    fn for_each_relevant_impl<R: VisitorResult>(
        self,
        trait_def_id: Self::TraitId,
        self_ty: Self::Ty,
        f: impl FnMut(Self::ImplId) -> R,
    ) -> R;
    fn for_each_blanket_impl<R: VisitorResult>(
        self,
        trait_def_id: Self::TraitId,
        f: impl FnMut(Self::ImplId) -> R,
    ) -> R;

    // === Trait properties ===
    fn trait_is_auto(self, trait_def_id: Self::TraitId) -> bool;
    fn trait_is_coinductive(self, trait_def_id: Self::TraitId) -> bool;
    fn trait_is_dyn_compatible(self, trait_def_id: Self::TraitId) -> bool;
    fn trait_is_fundamental(self, def_id: Self::TraitId) -> bool;
    fn impl_polarity(self, impl_def_id: Self::ImplId) -> ty::ImplPolarity;
    fn impl_specializes(self, a: Self::ImplId, b: Self::ImplId) -> bool;

    // === Lang items (Sized, Copy, Fn, etc.) ===
    fn require_trait_lang_item(self, lang_item: SolverTraitLangItem) -> Self::TraitId;
    fn require_projection_lang_item(self, lang_item: SolverProjectionLangItem)
        -> Self::TraitAssocTyId;

    // === Opaque types, coroutines, const evaluation ===
    fn opaque_types_defined_by(self, anchor: Self::LocalDefId) -> Self::LocalDefIds;
    fn coroutine_hidden_types(self, def_id: Self::CoroutineId) -> ...;
    fn fn_sig(self, def_id: Self::FunctionId) -> ty::EarlyBinder<...>;

    // ... ~100+ total associated types and methods
}

3. The SolverDelegate & InferCtxtLike Traits

Beyond Interner, you also need to implement the inference context and solver delegate:

InferCtxtLike — manages inference state

pub trait InferCtxtLike: Sized {
    type Interner: Interner;
    fn cx(&self) -> Self::Interner;

    // Typing mode (analysis vs coherence)
    fn typing_mode_raw(&self) -> TypingMode<Self::Interner>;

    // Universe management (for higher-ranked types)
    fn universe(&self) -> ty::UniverseIndex;
    fn create_next_universe(&self) -> ty::UniverseIndex;

    // Inference variable operations
    fn root_ty_var(&self, var: ty::TyVid) -> ty::TyVid;
    fn opportunistic_resolve_ty_var(&self, vid: ty::TyVid) -> Ty;
    fn next_region_infer(&self) -> Region;

    // Region constraint handling
    fn sub_regions(&self, sub: Region, sup: Region, ...);

    // Opaque type storage
    fn register_hidden_type_in_storage(&self, ...);
    fn clone_opaque_types_lookup_table(&self) -> Vec<...>;
}

SolverDelegate — solver callbacks

pub trait SolverDelegate: Deref<Target = Self::Infcx> + Sized {
    type Infcx: InferCtxtLike<Interner = Self::Interner>;
    type Interner: Interner;

    fn build_with_canonical<V>(
        cx: Self::Interner,
        canonical: &ty::CanonicalQueryInput<Self::Interner, V>,
    ) -> (Self, V, ty::CanonicalVarValues<Self::Interner>);

    fn compute_goal_fast_path(...) -> Option<Certainty>;
    fn leak_check(&self, max_input_universe: UniverseIndex) -> Result<(), NoSolution>;
    fn evaluate_const(...) -> Option<Const>;
    fn well_formed_goals(...) -> Option<Vec<Goal<...>>>;
    fn fetch_eligible_assoc_item(...) -> FetchEligibleAssocItemResponse<...>;
    fn is_transmutable(...) -> Result<Certainty, NoSolution>;
}

The SolverDelegate is manageable (~12 methods), but it assumes you have a full inference context with canonicalization support. The hard part isn't the delegate itself — it's building the InferCtxtLike beneath it.

4. Entry Points: How You Call the Solver

Once you've implemented the above, calling the solver is straightforward:

pub trait SolverDelegateEvalExt: SolverDelegate {
    /// Evaluates a goal from **outside** of the trait solver.
    fn evaluate_root_goal(
        &self,
        goal: Goal<Self::Interner, <Self::Interner as Interner>::Predicate>,
        span: <Self::Interner as Interner>::Span,
        stalled_on: Option<GoalStalledOn<Self::Interner>>,
    ) -> Result<GoalEvaluation<Self::Interner>, NoSolution>;

    /// Fast check: can this goal possibly hold?
    fn root_goal_may_hold_opaque_types_jank(
        &self,
        goal: Goal<Self::Interner, <Self::Interner as Interner>::Predicate>,
    ) -> bool;
}

A Goal is simply:

pub struct Goal<I: Interner, P> {
    pub param_env: I::ParamEnv,   // where clauses in scope
    pub predicate: P,             // what to prove
}

And predicates include:

enum ClauseKind<I: Interner> {
    Trait(TraitPredicate<I>),           // T: Trait
    Projection(ProjectionPredicate<I>), // <T as Trait>::Assoc == U
    TypeOutlives(...),                   // T: 'a
    RegionOutlives(...),                 // 'a: 'b
    ConstArgHasType(...),
    WellFormed(...),
    ConstEvaluatable(...),
}

enum PredicateKind<I: Interner> {
    Clause(ClauseKind<I>),
    NormalizesTo(NormalizesTo<I>),      // normalize <T as Trait>::Assoc
    AliasRelate(...),
    Subtype(SubtypePredicate<I>),
    Coerce(CoercePredicate<I>),
    Ambiguous,
}

5. SAGE Gap Analysis

Here's what SAGE currently has vs. what the solver needs:

What SAGE Has

  • Stash-allocated types with hash-consing (Ty<'db>, Ptr, Slice)
  • Binders and BoundVars for generic parameters
  • Type folding infrastructure (TyFolder, Instantiate)
  • Symbol resolution with module scoping
  • Trait and impl AST nodes with signature lowering
  • Salsa-based incremental computation
  • External crate metadata via TcxDb trait

What SAGE Is Missing

  • Where clause parsing and representation
  • Inference variables and unification
  • Universe management (for higher-ranked types)
  • Canonicalization (for caching queries across contexts)
  • Region/lifetime tracking (SAGE erases these)
  • Predicate/Clause IR types
  • ParamEnv (the set of assumptions in scope)
  • Impl indexing by self-type
  • Const evaluation

The Lifetime Problem

SAGE currently erases lifetimes. The rustc solver fundamentally requires them — Region is a core associated type on Interner, and region constraints are integral to the solving process. You'd need either:

  1. A stub Region type that acts as "always static" (lossy but workable for trait solving)
  2. Full lifetime representation (major scope expansion)

The Stash vs. Global Interning Problem

SAGE uses per-signature stash allocation. The rustc solver expects Copy types that are globally interned (one Interner value = one interning context). This is a fundamental mismatch:

  • Solver types must be Copy + Hash + Eq and valid across the entire solving session
  • SAGE's Ty<'db> is Copy but tied to a specific Stash lifetime
  • You'd need a global interning layer for types that participate in solving

6. Cost Comparison

Option A: Use rustc_next_trait_solver

Effort to integrate:

  • Implement Interner (~100+ assoc types/methods)
  • Implement InferCtxtLike (~20 methods)
  • Implement SolverDelegate (~12 methods)
  • Build canonicalization layer
  • Build global interning layer for solver types
  • Bridge SAGE's stash types ↔ solver types
  • Add predicate/clause IR
  • Handle lifetimes (even if stubbed)

Estimated: 8–12 weeks of focused work

Pro: Complete trait solving, specialization, coherence, normalization for free. Battle-tested on all of Rust.

Pro: Stays correct as the language evolves.

Con: Enormous surface area to implement and maintain.

Con: Couples SAGE to rustc's internal IR shape (which changes frequently, no stability guarantee).

Con: You get everything — coroutines, transmute checking, opaque types — whether you want it or not.

Option B: Write a Custom Solver

What you'd build:

  • Predicate/Clause types in SAGE's IR
  • ParamEnv (list of where-clause assumptions)
  • Impl candidate assembly (index by self-type)
  • Unification engine (type inference)
  • Goal evaluation with cycle detection
  • Normalization for associated types

Estimated: 4–8 weeks for a "good enough" solver

Pro: Fits SAGE's stash-allocated types natively.

Pro: Only implement what you need (no coroutines, no transmute, no opaque types).

Pro: No external dependency churn.

Pro: Can integrate with salsa directly for incremental trait solving.

Con: Must handle correctness yourself (coherence, specialization, negative impls).

Con: Risks diverging from Rust semantics on edge cases.

Con: Associated type normalization is surprisingly complex.

7. Assessment

Key Questions to Ask
  1. Does SAGE need to be sound for all Rust programs, or just "common" ones?
    If you need full soundness (specialization, coherence, negative reasoning), the rustc solver is the only reliable path. If you're okay with "works for 95% of code," a custom solver is viable.
  2. Does SAGE need lifetime-aware trait solving?
    If no (current state: lifetimes erased), a custom solver avoids the impedance mismatch entirely.
  3. How much does SAGE rely on external crate metadata?
    The TcxDb already provides crate data. If you route trait/impl info through TcxDb, you can feed either solver. But the rustc solver expects information in its own shape (EarlyBinder, Clause, etc.).
  4. How fast does trait solving need to be, and does it need incremental caching?
    The rustc solver's canonicalization enables cross-query caching. A salsa-backed custom solver gets incremental reuse at a different granularity.
Bottom Line

The rustc trait solver was designed to be reusable, but "reusable" here means "rust-analyzer could use it" — another project that already speaks rustc's type language. For SAGE, which has its own type representation, stash-based allocation, and no lifetime tracking, the integration cost is very high.

A hybrid approach may be best:

  1. Write a custom solver for SAGE's own trait/impl checking (local crate)
  2. Delegate to rustc (via TcxDb) for external crate trait queries where you need the "ground truth" answer about whether a type from an external crate satisfies a trait bound

This gets you: native integration with SAGE's IR, incremental caching via salsa, correctness for external crates (by asking rustc), and no need to implement the 100+ method Interner trait.
